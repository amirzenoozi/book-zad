# book-zad

A browser extension built on the same boilerplate as **bib-click**
(`../bib-click`). This file records the conventions inherited from that project —
stack, folder structure, package manager, and build/publish flow — so the new
extension stays consistent with it.

> **Status:** scoping. What the extension actually *does* is still TBD — see
> "Project scope" at the bottom. Everything above it is the fixed boilerplate.

## Stack

- **Framework:** [WXT](https://wxt.dev) — generates one manifest per browser from
  `wxt.config.ts` + `entrypoints/`. Chrome/Edge build as **MV3**, Firefox as
  **MV2** (WXT's default).
- **Language:** TypeScript. Type-check only, no emit (`tsc --noEmit`).
- **Package manager:** **npm** (`package-lock.json`, `npm ci` in CI). Not
  yarn/pnpm.
- **Messaging (optional):** `@webext-core/messaging` for typed
  background↔content↔popup messages (used in bib-click; keep if the new
  extension needs it).

## Folder structure

```
entrypoints/            # WXT entrypoints — each becomes part of the manifest
  background.ts         #   service worker (MV3) / event page (MV2)
  content/index.ts      #   content script (+ style.css)
  popup/index.html      #   toolbar popup (+ main.ts, style.css)
lib/                    # framework-agnostic logic, imported by entrypoints
icon/                   # extension + toolbar icons (16/32/48/128 png, mono svg)
public/                 # static assets copied verbatim into the build
docs/                   # GitHub Pages landing + privacy page
.github/workflows/      # ci.yml + release.yml
wxt.config.ts           # manifest: name, permissions, gecko id, data-collection
package.json            # scripts (the "actions") + deps
.output/                # build output + store zips (git-ignored)
```

## Actions (npm scripts)

Same script set as bib-click:

| Script | Does |
|---|---|
| `npm run dev` | Chrome dev build with HMR (opens a dev browser) |
| `npm run dev:firefox` | Firefox dev build with HMR |
| `npm run compile` | type-check (`tsc --noEmit`) |
| `npm run build` / `build:firefox` | production build per browser |
| `npm run zip` / `zip:firefox` / `zip:edge` | build + zip for a store |
| `npm run package` | zip **all three** stores in one go |
| `npm run submit:chrome` / `submit:firefox` / `submit:edge` | upload via `wxt submit` |

`postinstall` runs `wxt prepare` (generates `.wxt/` types) — so `npm install`
is enough to get a working tree.

## Build & publish flow

**Local package for the stores:**

```bash
npm run package         # builds + zips chrome, firefox, edge into .output/
```

Produces `book-zad-<version>-chrome.zip`, `-firefox.zip`, and a `-sources.zip`
(Firefox AMO requires the source bundle for review).

**Before the first publish:**

- Set a real extension id in `wxt.config.ts` →
  `browser_specific_settings.gecko.id` (currently placeholder).
- Bump `version` in `package.json` for each submission.
- Fill the store listing assets in `docs/` and `public/`.

**CI / release (GitHub Actions):**

- `ci.yml` — on push to `main` and every PR: `npm ci` → `compile` →
  `build` (Chrome) → `build:firefox`. Keeps the tree always-buildable.
- `release.yml` — on a pushed version tag (`v*`) or manual dispatch: type-check,
  zip all three stores, upload the zips as a workflow artifact, then publish via
  `wxt submit`. **Each store step is skipped unless its secrets are set**, so it
  works with Chrome-only, Firefox-only, or all three.

**Cutting a release:**

```bash
npm version patch       # or minor / major — updates package.json
git push && git push --tags
```

`release.yml` verifies the pushed tag matches `package.json` (`v0.2.0` ⇄
`0.2.0`) before publishing.

**Store secrets** (GitHub repo secrets, all optional per store):
`CHROME_EXTENSION_ID`, `CHROME_CLIENT_ID`, `CHROME_CLIENT_SECRET`,
`CHROME_REFRESH_TOKEN`; `FIREFOX_EXTENSION_ID`, `FIREFOX_JWT_ISSUER`,
`FIREFOX_JWT_SECRET`; `EDGE_PRODUCT_ID`, `EDGE_CLIENT_ID`, `EDGE_API_KEY`.
Run `npx wxt submit init` locally for the interactive walkthrough.

## Current state

**Scaffold built and green** — `npm run compile`, `npm run build` (Chrome MV3)
and `npm run build:firefox` (Firefox MV2) all pass. Icons are generated by
`npm run gen:icons` (a dependency-free PNG encoder in `scripts/gen-icons.mjs` —
they are **not** committed; CI regenerates them). See `README.md` for the file
map. What exists: background worker (opens manager, warms the folder index,
badge, visit recording), all-pages content script (page text + dwell tracking),
the full manager page, and `lib/` (bookmarks, storage, tfidf, similarity,
scoring, settings, theme, toast, text, messaging).

**Manager page** has two modes: a **folder view** (breadcrumb navigation
into/out of folders; create/rename/delete folders; move a bookmark via a
per-card "move to folder" dropdown) shown when the search box is empty, and a
flat **global search view** across all folders when it isn't. The last-open
folder is persisted (`storage.local` `ui.folder`) so a page refresh keeps your
place. Plus per-bookmark score (auto + manual stars), searchable notes, a
cleanup panel (duplicates + stale, with one-click "keep best, delete rest"
dedupe per group or across all groups), and a settings drawer.

Not yet done / rough edges: favicons are monogram tiles (deliberate — avoids
external requests); Firefox MV2 `browserAction` shim added but untested in a
live Firefox profile; the whole extension has not yet been exercised in a live
browser (only type-checked, unit-tested and built). Dead-link detection now
exists (best-effort background probe in the cleanup panel).

## Project scope

- **What it does:** a **bookmark manager** — manages the user's browser
  bookmarks "in a new way". Operates on browser bookmark data via the
  `bookmarks` API.
- **UI surface:** a **full manager page** (dedicated SPA) for all bookmark
  management, plus a small **toolbar popup** that acts as a launcher + quick
  settings. The toolbar icon opens the popup; its "Open manager" button opens
  the full page. (The popup exposes the nudge/toast toggles, the suggestion
  threshold, and theme; the manager also has its own settings drawer.)
- **Data model:** **layer metadata on native bookmarks.** The browser's real
  bookmarks stay the source of truth and remain usable outside the extension;
  we keep a parallel store (`storage`/IndexedDB) keyed by bookmark id for the
  extra fields the native API can't hold (notes, score, tags…).

### Features ("the new way")

1. **Powerful search** — full-text / fuzzy across title, URL, notes.
2. **Visual dashboard** — card grid with favicons/thumbnails, not a text tree.
3. **Notes + hygiene** — per-bookmark notes; dead-link detection, dedupe,
   "saved but never revisited" cleanup.
4. **Scoring & ranking** — every saved page gets a **score** and the dashboard
   can sort by it. **Both** automatic + manual: an auto baseline computed from
   usage signals (visit count, recency, dwell time — tracked locally by us, not
   via the `history` API), which the user can override with a manual rating.
5. **Similarity nudge** — while the user reads *any* page, detect that it's
   similar to something already saved in a folder and **notify them** to either
   add the current page to that folder, or revisit the already-saved documents
   there. **Runs automatically on every page** via a content script.

### Similarity engine (decided)

- **Local keyword / TF-IDF, fully on-device.** The content script extracts the
  current page's text; the background compares it (TF-IDF / keyword overlap)
  against the saved-bookmark index. **No external API, nothing leaves the
  device.** Weaker than embeddings on paraphrase, but private, tiny, and easy
  to ship past store review.
- Consequence: an **always-on content script on `<all_urls>`** + broad
  `host_permissions` (like bib-click). Privacy line for the listing: "reads the
  pages you visit to compare against your bookmarks; all processing is on-device
  and nothing is transmitted."

### Permissions (expected)

- `bookmarks` (core), `storage` (layered metadata, score data, TF-IDF index,
  settings).
- `tabs` / `activeTab` for the current page + quick-add.
- `host_permissions: ['<all_urls>']` + a content script — for the automatic
  similarity check on every page.
- **No** external API host and **no** `history` permission (usage tracked
  ourselves).
- No `notifications` permission by default — nudges use the toolbar badge
  (see below), which needs no extra permission.

### Nudge delivery (decided)

- **Default: toolbar badge.** When the current page matches a saved folder, set
  a badge on the extension icon (`action.setBadgeText` / `setBadgeBackgroundColor`)
  — quiet, permission-free, dismissible by clicking through to the manager.
- **Optional: in-page toast**, enabled in settings. When on, the content script
  shows a small in-page toast in addition to the badge. Off by default to keep
  browsing uninterrupted. The toast is **persistent + actionable**: it stays
  until the user closes it or acts, with **"+ Add to <folder>"** (creates the
  bookmark in the matched folder via `addToFolder` → background) and **"Open
  folder"** (`openManagerAt` → opens the manager focused on that folder).

### Theming (decided)

- **Dark / light / auto** themes across the manager page (and toast). "Auto"
  follows the OS via `prefers-color-scheme`; the manual choice is persisted in
  settings (`storage`). Design the CSS theme-aware from the start.
- **Palette adapted from `../pingsx-overview`** (`modal.css`): deep-navy panels,
  teal accent (`#46bea0` dark / `#1f9e86` light), amber/red for warn/danger,
  monospace (`--mono`) for numerics (scores, stats, urls, counts), uppercase
  letter-spaced micro-labels, pill chips. Badge + toast + generated icons all
  use the teal accent to match.
- **Persian/Arabic text uses Vazirmatn**, bundled locally in `public/fonts/`
  (arabic subset, weights 400/500/600/700) — never fetched from a CDN, per the
  privacy stance. Scoped by `unicode-range` (in `lib/fonts.css`) so only
  Persian/Arabic glyphs use it and Latin keeps the system font. Manager + popup
  import `lib/fonts.css`; the toast (shadow DOM on host pages) injects its own
  `@font-face` via `browser.runtime.getURL`, so the fonts are declared in
  `web_accessible_resources`.

### Localisation (decided)

- **`_locales` + the `i18n` API**, not Chrome's page translation (which never
  fires on `chrome-extension://` pages, nor inside the toast's shadow root).
  Messages live in `public/_locales/<lang>/messages.json`; **en** is
  `default_locale` and **fa** ships alongside it. The browser picks by UI
  language and falls back to English.
- The **manifest** is localised too (`__MSG_ext_name__`,
  `__MSG_ext_description__`), which also localises the Chrome Web Store listing.
  Keep each locale's `ext_description` **≤132 chars**.
- `lib/i18n.ts` is the only entry point: `t()` for code, `data-i18n` /
  `data-i18n-placeholder` / `data-i18n-title` attributes for markup (resolved by
  `localizeDom()`), and `plural()` because the i18n API has no plural rules.
  `t()` is typed against the key union WXT generates from `messages.json`, so a
  bad key is a **compile error** — after editing messages, run `wxt prepare`.
- **RTL**: `applyDirection()` stamps `<html lang|dir>` from the active locale
  (`locale_dir` in each messages file). Layout uses **logical properties**
  (`margin-inline-start`, `inset-inline-end`, `text-align: start`) rather than a
  second stylesheet; only genuinely non-logical bits (drawer shadow, accordion
  chevron) sit behind `[dir='rtl']`. User data (titles, folder names, URLs) gets
  `unicode-bidi: plaintext` so each string takes direction from its own content.
- Persian/Arabic glyphs render in **Vazirmatn** automatically via the
  unicode-range `@font-face` in `lib/fonts.css` — no per-locale font switch
  needed, and Latin keeps the system font.
